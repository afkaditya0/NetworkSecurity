Message Authentication

Message Authentication is concerned with: protecting the integrity of a message, validating identity of originator, and non-repudiation of origin (dispute resolution).
Three Alternative Functions
Hash Function
⟶
Message Encryption
⟶
Message Authentication Code (MAC)
Message Security Requirements
Passive Threats

What attackers observe

  • Disclosure — release of message contents
  • Traffic analysis — patterns of communication
Active Threats

What attackers do

  • Masquerade — fake source identity
  • Content modification
  • Sequence modification
  • Timing modification — replay/delay
Repudiation

Denials

  • Source repudiation — sender denies sending
  • Destination repudiation — receiver denies receiving
Encryption-Based Authentication

Symmetric Encryption

  • Receiver knows sender must have created it
  • Only sender and receiver know the key
  • Need suitable structure / redundancy / checksum to detect changes

Public-Key Encryption

  • Encryption alone gives no sender confidence (public key is, well, public)
  • Sender signs with private key, then encrypts with recipient's public key → secrecy + authentication
  • Costs two public-key uses per message
02

Message Authentication Code (MAC)

★ Definition: A small fixed-sized block generated from message + secret key.
MAC = C(K, M)
Appended to message; receiver re-computes and compares. Provides assurance that the message is unaltered and comes from the sender. A MAC is NOT a digital signature — it provides authentication but not non-repudiation.
Properties
Cryptographic Checksum

MAC = CK(M)

  • Condenses a variable-length message M
  • Using a secret key K
  • To a fixed-sized authenticator
  • Many-to-one function (collisions exist but must be hard to find)
Why MAC?

Use cases

  • Sometimes only authentication is needed — no secrecy
  • Authentication may need to persist longer than encryption (e.g., archival)
  • Generally MAC is computed before encryption when both used
Need not be reversible

Like encryption, but…

  • Computation depends on both message and key
  • Unlike encryption, MAC need not be reversible
  • Generally use separate keys for encryption vs. MAC
Requirements for MACs
R1Forgery resistance — knowing a message and its MAC, infeasible to find another message with the same MAC
R2Uniform distribution — MACs should be uniformly distributed
R3All-bit dependence — MAC should depend equally on all bits of the message
Security of MACs
Brute Force

Two attack surfaces

  • Strong collision resistance hash has cost 2m/2
  • 128-bit hash looks vulnerable, 160-bit better
  • Can attack the keyspace (cf. key search) or the MAC
  • At least 128-bit MAC is needed for security
Cryptanalysis

Structural attacks

  • Like block ciphers, want brute force to be the best alternative
  • More variety of MACs → harder to generalize
03

Keyed Hash Functions — HMAC

★ HMAC is the dominant MAC construction — Internet standard RFC 2104. It builds a MAC from any cryptographic hash function (MD5, SHA-1, RIPEMD-160, Whirlpool, …). Used in IPsec, TLS, SSH and many other protocols.
Why Keyed Hashes?
Motivation

Hash-based MAC

  • Hash functions are generally faster than block ciphers
  • Crypto hash function code is widely available
  • Original naive proposal: KeyedHash = Hash(Key | Message)
  • Some weaknesses found → led to HMAC
Length-Extension Attack

Why naive Hash(K|M) fails

M = Key | Message | Padding M = M₁ M₂ … M_L (each block = blocksize) Hash = H(H(…H(H(IV,M₁),M₂)…),M_L) But attacker can append extra block M_{L+1}: Hash' = H(Hash, M_{L+1}) without knowing the key!
★ HMAC Design Objectives
D1Use, without modifications, existing hash functions
D2Allow easy replaceability of the embedded hash function
D3Preserve original performance of hash function without significant degradation
D4Use and handle keys in a simple way
D5Have well understood cryptographic analysis of authentication strength
★ HMAC Construction
HMAC_K(M) = Hash[ (K⁺ ⊕ opad) || Hash[ (K⁺ ⊕ ipad) || M ] ] where: K⁺ = key padded out to block size ipad = 0x36 repeated (inner pad) opad = 0x5C repeated (outer pad) Overhead: just 3 extra hash block calculations beyond what the message itself needs.
HMAC Security
Provable Security

Tied to underlying hash

  • Proved security of HMAC relates to that of the underlying hash algorithm
  • Attacking HMAC requires either:
    • Brute-force on the key, OR
    • Birthday attack — but since keyed, attacker would need a very large number of messages
  • Choose hash based on speed vs. security tradeoff
04

Symmetric Cipher MACs — DAA & CMAC

Any block cipher chaining mode can be used: encrypt the message and use the final block as a MAC.
Data Authentication Algorithm (DAA)
DAA

DES-CBC based MAC

  • Widely used MAC based on DES-CBC
  • Use IV = 0 and zero-pad of final block
  • Encrypt message using DES in CBC mode
  • Send just the final block (or leftmost M bits, 16 ≤ M ≤ 64) as the MAC
Final MAC is now too small for security — and message blocks could be reordered. Hence the move to CMAC.
CMAC

Cipher-based MAC (NIST SP 800-38B)

  • Overcomes DAA's message-size limitation
  • Uses 2 keys & padding
  • Adopted as a standard by NIST
  • Widely used in government & industry
05

Authenticated Encryption

Simultaneously protect confidentiality AND authenticity of communications. Often required, but historically done separately — leading to subtle bugs.
Four Composition Approaches
Approach A

Hash-then-Encrypt

E(K, (M || H(M)))
Approach B

MAC-then-Encrypt

E(K₂, (M || MAC(K₁, M)))
Approach C

Encrypt-then-MAC

C = E(K₂, M), T = MAC(K₁, C)
Approach D

Encrypt-and-MAC

C = E(K₂, M), T = MAC(K₁, M)
Decryption / verification is straightforward — but each composition has subtle security vulnerabilities. Modern protocols use dedicated AEAD modes like CCM or GCM.
★ CCM — Counter with CBC-MAC
NIST SP 800-38C

Used in WiFi (WPA2)

  • Variation of encrypt-and-MAC approach
  • Algorithmic ingredients:
    • AES encryption algorithm
    • CTR mode of operation
    • CMAC authentication algorithm
  • Single key used for both encryption & MAC
GCM — NIST SP 800-38D

Galois / Counter Mode

  • Parallelizable — high performance
  • Message encrypted in variant of CTR
  • Ciphertext multiplied with key & length over GF(2¹²⁸) to generate authenticator tag
  • GMAC — MAC-only mode also defined
  • Uses two functions: GHASH (keyed hash) and GCTR (CTR with incremented counter)

PRNG using Hash & MAC

Essential elements of any PRNG: a seed value + a deterministic algorithm. The seed must be known only as needed.
Hash-based PRNG

SP800-90 / ISO 18031

Take seed V Repeat: V = V + 1 output = n-bits of Hash(V)

Secure if a good hash is used.

MAC-based PRNG

SP800-90, IEEE 802.11i, TLS

  • Use a key K
  • Input based on the last hash in various ways
  • Output drives session-key generation

Digital Signatures

★ Why Digital Signatures? Message authentication does NOT address lack of trust. Digital signatures additionally provide:
  • Verify author, date & time of signature
  • Authenticate message contents
  • Be verified by third parties to resolve disputes (non-repudiation)
Attacks on Signatures
Attack Models

Five attack types

  • Key-only attack
  • Known message attack
  • Generic chosen message attack
  • Directed chosen message attack
  • Adaptive chosen message attack
Break Levels

Success severity

  • Total break — recover signing key
  • Selective forgery — forge for chosen msg
  • Existential forgery — forge for some msg
★ Signature Requirements
S1Depend on the message signed
S2Use info unique to sender — to prevent both forgery and denial
S3Relatively easy to produce
S4Relatively easy to recognize & verify
S5Computationally infeasible to forge — both with new message for existing signature, and with fraudulent signature for given message
S6Practical to save in storage
Direct Digital Signatures
Two-Party Scheme

Sender ↔ Receiver only

  • Assumes receiver has sender's public key
  • Signature made by sender signing entire message (or its hash) with private key
  • Can encrypt using receiver's public key for confidentiality
  • Important: sign first, then encrypt the message + signature
  • Security depends entirely on sender's private key

ElGamal & Schnorr Signatures

★ ElGamal Digital Signature
Signature variant of ElGamal, related to D-H. Uses exponentiation in a finite (Galois) field. Security based on difficulty of computing discrete logarithms. Uses private key for signing, public key for verification.
Key Generation
Each user A: Chooses secret key: 1 < x_A < q − 1 Computes public key: y_A = a^(x_A) mod q
Signing
Alice signs message M to Bob: 1. Compute hash: m = H(M), 0 ≤ m ≤ q−1 2. Choose random K: 1 ≤ K ≤ q−1, gcd(K, q−1) = 1 3. Temporary key: S₁ = a^K mod q 4. Compute K⁻¹ mod (q−1) 5. S₂ = K⁻¹ · (m − x_A · S₁) mod (q−1) 6. Signature is the pair: (S₁, S₂)
Verification
Any user B verifies by computing: V₁ = a^m mod q V₂ = (y_A)^S₁ · (S₁)^S₂ mod q Signature valid iff V₁ = V₂
Worked Example — GF(19), a = 10
Alice's key: x_A = 16 → y_A = 10¹⁶ mod 19 = 4 Sign message with hash m = 14: Random K = 5 (gcd(18, 5) = 1) S₁ = 10⁵ mod 19 = 3 K⁻¹ mod 18 = 5⁻¹ mod 18 = 11 S₂ = 11 · (14 − 16·3) mod 18 = 4 Signature: (3, 4) Verification: V₁ = 10¹⁴ mod 19 = 16 V₂ = 4³ · 3⁴ = 5184 mod 19 = 16 ✓
★ Schnorr Digital Signatures
Schnorr Setup

Properties

  • Also uses exponentiation in a finite (Galois) field
  • Security based on discrete logarithms
  • Minimizes message-dependent computation — multiplying a 2n-bit integer with an n-bit integer
  • Main work can be done in idle time (precomputation)
  • Prime modulus p; p−1 has prime factor q (typically p = 1024-bit, q = 160-bit)
Key Setup

Schnorr Keys

Choose primes p, q where q | (p−1) Choose a such that a^q = 1 mod p (a, p, q) are global parameters. Each user A: Secret key: 0 < s_A < q Public key: v_A = a^(−s_A) mod q
Schnorr Signing & Verification
SIGN: Choose random r, 0 < r < q x = a^r mod p e = H(M || x) y = (r + s·e) mod q Signature = (e, y) VERIFY: x' = a^y · v^e mod p Check: e == H(M || x')

Digital Signature Standard (DSS / DSA)

★ DSS — US Govt approved signature scheme. Designed by NIST & NSA in early 90s. Published as FIPS-186 in 1991, revised 1993, 1996, 2000. Uses the SHA hash algorithm. DSS is the standard, DSA is the algorithm. FIPS 186-2 (2000) added RSA & ECC variants.
DSA Properties

At a glance

  • Creates a 320-bit signature
  • With 512–1024 bit security
  • Smaller and faster than RSA
  • Digital signature scheme only — unlike RSA which also encrypts
  • Public-key technique
  • Security depends on difficulty of discrete logarithms
  • A variant of ElGamal & Schnorr schemes
DSS vs RSA

Key difference

  • RSA: signature = encryption with private key, verification = decryption with public key
  • DSS: signature uses random k and the message hash; verification uses a different equation
  • DSS cannot be used to encrypt — only sign
★ DSA Key Generation
Global public key (p, q, g): Choose 160-bit prime q Choose large prime p with 2^(L−1) < p < 2^L where L = 512–1024, multiple of 64 such that q is a 160-bit prime divisor of (p−1) Choose g = h^((p−1)/q) mod p where 1 < h < p−1 and g > 1 Each user: Private key: random x < q Public key: y = g^x mod p
★ DSA Signature Creation
To sign message M, sender: Generates random signature key k, k < q ⚠️ k must be random, destroyed after use, and NEVER reused Compute signature pair: r = (g^k mod p) mod q s = [k⁻¹ (H(M) + x·r)] mod q Send (r, s) with message M
★ DSA Signature Verification
Receiver has M, (r, s): w = s⁻¹ mod q u₁ = [H(M) · w] mod q u₂ = (r · w) mod q v = [(g^u₁ · y^u₂) mod p] mod q Signature valid iff v = r
Critical: reuse of k across two different messages allows recovery of the private key. PlayStation 3 was famously broken because Sony reused k.

Kerberos

★ Kerberos — trusted key server system from MIT. Provides centralised private-key third-party authentication in a distributed network. Allows users access to services distributed through a network without needing to trust all workstations — instead, all parties trust a central authentication server. Two versions in active use: v4 & v5.
Kerberos Requirements
Req 1

Secure

Resist eavesdropping and impersonation.

Req 2

Reliable

Service must be highly available — KDC failure = no logins.

Req 3

Transparent

User shouldn't notice authentication beyond entering a password.

Req 4

Scalable

Support many clients and servers across realms.

★ Kerberos v4 — Two Servers
AS — Authentication Server

Initial credential

  • Users initially negotiate with AS to identify themselves
  • AS provides a non-corruptible authentication credential — the Ticket Granting Ticket (TGT)
TGS — Ticket Granting Server

Service tickets

  • Users subsequently request access to other services from the TGS
  • TGS issues service tickets on the basis of the user's TGT
  • Uses a complex protocol with DES
Kerberos v4 Dialogue (Simplified)
1. C → AS: ID_C, ID_TGS, TS₁ 2. AS → C: E(K_C, [K_{C,TGS}, ID_TGS, TS₂, Lifetime, Ticket_TGS]) where Ticket_TGS = E(K_TGS, [K_{C,TGS}, ID_C, AD_C, ID_TGS, TS₂, Lifetime]) 3. C → TGS: ID_V, Ticket_TGS, Authenticator_C 4. TGS → C: E(K_{C,TGS}, [K_{C,V}, ID_V, TS₄, Ticket_V]) 5. C → V: Ticket_V, Authenticator_C 6. V → C: E(K_{C,V}, TS₅ + 1) (mutual authentication)
Kerberos Realms
Realm Definition

Single administrative domain

  • A Kerberos environment consists of: a Kerberos server, registered clients, and application servers sharing keys with the server
  • This is termed a realm
  • Typically a single administrative domain
  • Multiple realms must share keys and trust between their Kerberos servers
Kerberos Version 5

Improvements over v4

  • Developed mid-1990s, standardised as RFC 1510
  • Environmental fixes: encryption alg, network protocol, byte order, ticket lifetime, authentication forwarding, inter-realm auth
  • Technical fixes: double encryption, non-standard mode of use, session keys, password attacks

X.509 Certificates

★ Public-Key Certificates allow key exchange without real-time access to a public-key authority. A certificate binds identity to public key, with all contents signed by a trusted Certificate Authority (CA). Anyone who knows the CA's public key can verify it.
X.509 Authentication Service
Origin

CCITT X.500

  • Part of CCITT X.500 directory service standards
  • Distributed servers maintaining a user-info database
  • Defines a framework for authentication services
  • Directory may store public-key certificates with user public key signed by CA
  • Also defines authentication protocols
  • Uses public-key crypto & digital signatures (RSA recommended)
  • X.509 certs are widely used; have 3 versions
★ X.509 Certificate Fields

SignatureHash of all fields, signed by CA

Notation: CA<<A>> denotes a certificate for A signed by CA.

Obtaining a Certificate
Public Distribution

Properties

  • Any user with access to the CA can get any certificate from it
  • Only the CA can modify a certificate
  • Because they cannot be forged, certificates can be placed in a public directory
CA Hierarchy

Cross-CA verification

  • If both users share a common CA they're assumed to know its public key
  • Otherwise CAs must form a hierarchy
  • Use certificates linking members of hierarchy to validate other CAs
  • Each CA has certificates for clients (forward) and parent (backward)
  • Each client trusts parent's certificates
  • Enables verification of any cert from one CA by users of all other CAs in the hierarchy
Certificate Revocation
Certificates have a period of validity — but may need to be revoked before expiry:
  1. User's private key is compromised
  2. User is no longer certified by this CA
  3. CA's certificate is itself compromised
CAs maintain a Certificate Revocation List (CRL). Users should always check certificates against the CA's CRL.
X.509 Version 3 Extensions
Why v3?

Generic extension method

  • Additional info needed: email/URL, policy details, usage constraints
  • Rather than naming new fields explicitly, defined a general extension method
  • Each extension consists of: identifier, criticality indicator, value
Extension Categories

Three groups

  • Key & policy info — about subject & issuer keys, certificate policy
  • Subject & issuer attributes — alternative names/formats
  • Certification path constraints — limit how other CAs can use this cert

Public Key Infrastructure (PKI)

PKI is the set of hardware, software, people, policies and procedures needed to create, manage, store, distribute and revoke digital certificates based on asymmetric crypto.
PKIX Management Functions
F1

Registration

Bind user identity to certificate request.

F2

Initialization

Establish trust anchors and initial keys.

F3

Certification

CA issues certificate after verifying request.

F4

Key Pair Recovery

Restore lost decryption keys (escrow).

F5

Key Pair Update

Periodic re-keying for active certificates.

F6

Revocation Request

User or CA flags certificate as no-longer-valid.

F7

Cross Certification

Trust between distinct CAs/realms.

Management protocols: CMP (Certificate Management Protocol) and CMC (Certificate Management over CMS).
