★ SSL Handshake Protocol — 4 Phases
Allows server & client to: authenticate each other, negotiate encryption & MAC algorithms, and negotiate cryptographic keys. Comprises a series of messages in 4 phases.
01

Establish Security Capabilities

client_hello ↔ server_hello — protocol version, session ID, cipher suite, compression method, initial random numbers.

02

Server Authentication & Key Exchange

Server may send certificate, server_key_exchange, certificate_request, then server_hello_done.

03

Client Authentication & Key Exchange

Client sends certificate if requested, client_key_exchange, optionally certificate_verify.

04

Finish

Both sides: change_cipher_spec then finished. Cipher suite changes; handshake done.

Handshake message format: [ Type (1B) ][ Length (3B) ][ Content (≥0B) ]
★ TLS — Transport Layer Security
IETF Standard RFC 2246

Similar to SSLv3, with minor differences

  • Different record-format version number
  • Uses HMAC for MAC
  • A pseudo-random function expands secrets (based on HMAC using SHA-1 or MD5)

Secure Shell (SSH)

★ SSH — protocol for secure network communications, designed to be simple & inexpensive. SSH1 provided secure remote logon — replaced TELNET and other insecure schemes. SSH2 fixes a number of security flaws. Documented in RFCs 4250–4254. Method of choice for remote login and X tunnels.

Based on host key pairs

  • Server authentication occurs at the transport layer, based on server/host key pair(s)
  • Server authentication requires clients to know host keys in advance
Packet Exchange

  • Establish TCP connection
  • Then exchange data:
    • Identification string exchange
    • Algorithm negotiation
    • Key exchange
    • End of key exchange
    • Service request
  • Using specified packet format

  • SSH_MSG_USERAUTH_SUCCESS

  • Public-key
  • Password

SSH Connection Protocol
Overview

Multiplexed channels

  • Runs on SSH Transport Layer Protocol
  • Assumes a secure authentication connection
  • Used for multiple logical channels — SSH communications use separate channels
  • Either side can open a channel with unique id number
  • Flow-controlled
Channel Lifecycle

Three stages & four types

  • Stages: opening a channel → data transfer → closing a channel
  • Types: session, x11, forwarded-tcpip, direct-tcpip

★ Port Forwarding (SSH Tunneling)
Core Idea

Convert insecure TCP → secure SSH

  • SSH Transport Layer Protocol establishes a TCP connection between SSH client & server
  • Client traffic redirected to local SSH, travels via tunnel, then remote SSH delivers to server
Two Types

Local vs Remote

  • Local forwarding — hijacks selected traffic from the local end
  • Remote forwarding — client acts for server (reverse)

Secure Electronic Transactions (SET)

★ SET — an open encryption & security specification to protect Internet credit card transactions. Developed in 1996 by MasterCard, Visa et al. Note: SET is not a payment system — rather it is a set of security protocols & formats enabling:
  • Secure communications amongst parties
  • Trust from use of X.509v3 certificates
  • Privacy by restricting info to those who need it
SET Components
Party 1

Cardholder

The customer with a certified credit card.

Party 2

Merchant

Sells goods/services; has own SET certificate.

Party 3

Issuer

Bank that issued the cardholder's credit card.

Party 4

Acquirer

Merchant's bank — processes payments.

Party 5

Payment Gateway

Bridges merchant and existing bank payment network.

Party 6

Certificate Authority

Issues X.509v3 certificates to all parties.

04

05

06

08

09

10

Capture from payment gateway.

★ Dual Signature — The Heart of SET
Customer creates dual messages:
  • Order Information (OI) — for merchant
  • Payment Information (PI) — for bank
Neither party needs details of the other, but must know they are linked. Use a dual signature for this — signed concatenated hashes of OI & PI.
Dual signature construction: POMD = H( H(PI) || H(OI) ) DS = E(PR_c, POMD) // signed with customer's private key Merchant receives: PIMD, OI, DS, cert(PU_c) Compute POMD_m = H( PIMD || H(OI) ) Verify D(PU_c, DS) == POMD_m ✓ (order not tampered) Payment gateway receives: PI, OIMD, DS, cert(PU_c) Compute POMD_b = H( H(PI) || OIMD ) Verify D(PU_c, DS) == POMD_b ✓ (payment not tampered)
Purchase Request — Merchant Processing
01

Verify cardholder certificates

Using CA signatures.

02

Verify dual signature

Using customer's public signature key — ensures order not tampered with in transit, and was signed with cardholder's private signature key.

03

Process order & forward payment info

To the payment gateway for authorization.

04

Send purchase response

To the cardholder.

Payment Gateway Authorization — 8 Steps
01

Verify all certificates

02

Decrypt digital envelope of auth block

Obtain symmetric key, then decrypt authorization block.

03

Verify merchant's signature on auth block

04

Decrypt digital envelope of payment block

Obtain symmetric key, decrypt payment block.

05

Verify dual signature on payment block

06

Verify transaction ID match

Between merchant's and customer's PI.

07

08

Payment Capture

Funds transfer

  • Gateway checks request
  • Then causes funds to be transferred to merchant's account
  • Notifies merchant using capture response

Email Security & PGP

Email is one of the most widely used and regarded network services, but currently message contents are not secure — may be inspected in transit or by privileged users on the destination system.
Email Security Enhancements — Four Needs
N1

Confidentiality

Protection from disclosure.

N2

Authentication

Of sender of message.

N3

Message Integrity

Protection from modification.

N4

Non-repudiation of origin

Protection from denial by sender.

★ Pretty Good Privacy (PGP)
PGP — widely used de facto secure email. Developed by Phil Zimmermann. Selected best available crypto algorithms, integrated into a single program. Runs on Unix, PC, Macintosh and other systems. Originally free; now also has commercial versions.
PGP Operations
Authentication

Signed hash with RSA

  1. Sender creates message
  2. Make SHA-1 160-bit hash of message
  3. Attach RSA-signed hash to message
  4. Receiver decrypts & recovers hash code
  5. Receiver verifies received message hash
Confidentiality

Session key with RSA envelope

  1. Sender forms 128-bit random session key
  2. Encrypts message with session key
  3. Attaches session key encrypted with RSA
  4. Receiver decrypts & recovers session key
  5. Session key is used to decrypt message
Confidentiality + Authentication Combined
Can use both services on the same message:
  1. Create signature & attach to message
  2. Encrypt both message & signature
  3. Attach RSA/ElGamal-encrypted session key
Compression & Email Compatibility
Compression

After signing, before encrypting

  • By default PGP compresses message after signing but before encrypting
  • So can store uncompressed message & signature for later verification
  • Also because compression is non-deterministic
  • Uses ZIP compression algorithm
Email Compatibility

Radix-64 encoding

  • Encrypted data is binary, but email was designed only for text
  • PGP encodes raw binary data into printable ASCII
  • Uses radix-64 algorithm — maps 3 bytes to 4 printable chars
  • Also appends a CRC
  • PGP segments messages if too big
PGP Generic Transmission & Reception
TRANSMISSION (from A): X ← file if signature required: X ← signature || X X ← Z(X) // compress if confidentiality required: X ← E(PU_b, K_s) || E(K_s, X) // encrypt X ← R64[X] // radix-64 RECEPTION (to B): X ← R64⁻¹[X] if confidentiality: decrypt session key, decrypt X X ← Z⁻¹(X) // decompress if signature: strip signature, verify
PGP Session Keys
Per-message Keys

Fresh each time

  • Need a session key for each message
  • Varying sizes: 56-bit DES, 128-bit CAST or IDEA, 168-bit Triple-DES
  • Generated using ANSI X12.17 mode
  • Uses random inputs from previous uses and keystroke timing of user
Key IDs

Identify which public/private key

  • Many public/private keys may be in use — need to identify which was used
  • Sending full public key with every message is inefficient
  • Use a key identifier = least significant 64 bits of the key (very likely unique)
  • Also used in signatures
★ PGP Message Format
┌───────────────────┐ │ Session Key comp │ [KeyID(PU_b)] [Session Key K_s] encrypted with E(PU_b, ·) ├───────────────────┤ │ Signature │ [Timestamp] [KeyID(PU_a)] [Leading 2B MD] │ │ [Message Digest] signed with E(PR_a, ·) ├───────────────────┤ │ Message │ [Filename] [Timestamp] [Data] └───────────────────┘ The whole is ZIP-compressed, then encrypted with session key K_s, then radix-64 encoded for transport.
★ PGP Key Rings
Private Key Ring

Owner's keys

  • Contains the public/private key pair(s) for this user
  • Indexed by Key ID
  • Private key is encrypted under a hashed passphrase
  • Fields: Timestamp, Key ID, Public Key, Encrypted Private Key, User ID
Public Key Ring

All known peers

  • Contains public keys of other PGP users known to this user
  • Indexed by Key ID
  • Fields: Timestamp, Key ID, Public Key, Owner Trust, User ID, Key Legitimacy, Signature(s), Signature Trust(s)
Security of private keys depends entirely on the passphrase security.
★ PGP Key Management — Web of Trust
Decentralized Trust

Every user is own CA

  • Rather than relying on certificate authorities
  • In PGP every user is their own CA
  • Can sign keys for users they know directly
  • Forms a "web of trust"
  • Trust keys you have signed
  • Can trust keys others have signed, if there is a chain of signatures to them
  • Key ring includes trust indicators
  • Users can also revoke their keys
Trust Model Example

Legend

  • ? = unknown signatory
  • X → Y = X is signed by Y
  • Filled circle = owner is trusted by you to sign keys
  • Partial fill = partially trusted to sign
  • Dot = key is deemed legitimate by you

S/MIME

★ S/MIME — Secure/Multipurpose Internet Mail Extensions. A security enhancement to MIME email.
  • Original Internet RFC822 email was text-only
  • MIME provided support for varying content types and multi-part messages
  • With encoding of binary data to textual form
  • S/MIME added security enhancements
Supported in many mail agents: MS Outlook, Mozilla, Mac Mail, etc.
S/MIME Functions — 4 Types

Enveloped Data

Encrypted content and associated keys.

F2

Signed Data

Encoded message + signed digest.

F3

Clear-signed Data

Cleartext message + encoded signed digest (readable without S/MIME).

F4

Signed & Enveloped Data

Nesting of signed & encrypted entities.

Messages

MIME-wrapped PKCS objects

  • Secures a MIME entity with a signature, encryption, or both
  • Forms a MIME-wrapped PKCS object
  • Content-types: enveloped data, signed data, clear-signed data, registration request, certificate-only message
Certificate Processing

Hybrid X.509 + web of trust

  • And own public/private key pairs & certs

DomainKeys Identified Mail (DKIM)

DKIM — a specification for cryptographically signing email messages, so the signing domain claims responsibility. Recipients / agents can verify the signature. Proposed Internet Standard RFC 4871. Has been widely adopted.
Internet Mail Architecture
MUA (author) ──SMTP──► MSA ──SMTP──► MTA ──SMTP──► MTA ──► MDA ──► MS ──► MUA (recipient) │ (IMAP/POP) MUA = Message User Agent, MSA = Mail Submission Agent MTA = Message Transfer Agent, MDA = Mail Delivery Agent, MS = Message Store
Email Threats (RFC 4684)
Range

Who attacks

  • Low-end attackers
  • Spammers
  • Fraudsters
Capabilities

Attack vectors

  • Where submitted
  • Signed messages
  • Volume, routing, naming
Location

Network position

  • Outside-located attackers
DKIM Strategy
Transparent to User

MSA signs, MDA verifies

  • MSA signs outgoing mail with private key
  • MDA verifies via DNS lookup of signer's public key
  • For pragmatic reasons: users don't have to manage keys
Functional Flow

Sign → transit → verify → assess

  • RFC 5322 message signed with SDID at originating/relaying ADMD
  • Private key store at the signer
  • Public key store in DNS at the verifier
  • Message signed? → Verify signature → Pass or Fail
  • Check signing practices (reputation, local info)
  • Feed to message filtering engine

Intruders & IDS

A significant issue for networked systems is hostile or unwanted access — either via network or local. Can identify classes of intruders with varying levels of competence.
★ Types of Intruders (Anderson)
Class 1

Masquerader

Not authorized to use the computer. Penetrates a system's access controls to exploit a legitimate user's account. Usually external.

Class 2

Misfeasor

A legitimate user who accesses data/programs/resources not authorized, OR who is authorized but misuses their privileges. Usually internal.

Class 3

Clandestine User

Seizes supervisory control of the system and uses it to evade auditing and access controls, or to suppress audit actions. Either internal or external.

What is an Intrusion?
Any set of actions that attempt to compromise the confidentiality, integrity, or availability of a computer resource.
Consequences of Intrusion
Benign end

Exploration

Many people simply wish to explore the Internet.

Serious end

Damage

  • Read privileged data
  • Perform unauthorized modification
  • Disrupt system settings
★ Intrusion Detection Systems (IDS)
What IDS Does

Detect & respond

  • Intrusion detection = process of identifying and responding to malicious activity targeted at resources
  • IDS tests/analyses network system traffic/events against a given set of parameters and alerts/captures data when thresholds are met
  • Uses collected information + predefined knowledge-based system to reason about the possibility of an intrusion
  • Provides services to cope with intrusion: alarms, activating programs, etc.
Why IDS?

Firewalls alone are insufficient

  • Anti-virus only detects known viruses
  • Passwords can be hacked/stolen
  • Firewalls DO NOT recognize attacks and block them — just a fence
  • ~80% of financial losses from insider attacks — beware internal intruders
  • Example: Apr 1999, many sites hacked via a ColdFusion bug, despite firewalls allowing port 80
Functions & Principles of IDS
Functions

Reactive, not proactive

  • Detect attacks as soon as possible and take appropriate action
  • Does not usually take preventive measures when attack is detected
  • Reactive rather than pro-active agent
  • Plays role of informant rather than police officer
Principles

Deployment requirements

  • Must run unattended for extended periods
  • Must stay active and secure
  • Must recognize unusual activity
  • Must operate without unduly affecting the system
  • Must be configurable
Typical Intrusion Scenario
01

Information Gathering

Find as much info as possible: whois lookup, DNS zone transfers, normal browsing.

02

Further Information Gathering

Ping sweeps, port scanning, web server vulnerabilities, version probing of applications/services.

03

Attack!

Start trying different attacks. Find misconfigured services. Passive or active attack.

04

Successful Intrusion

Install own backdoors, delete log files. Replace services with Trojan horses with backdoor passwords; create own user accounts.

05

Fun and Profit

Steal confidential information. Use compromised host to launch further attacks. Change website for FUN.

Intrusion Techniques
Methodology

Four-phase attack

  • Target acquisition & information gathering
  • Initial access
  • Privilege escalation
  • Covering tracks

Key goal is often to acquire passwords → then exercise access rights of owner.

Password Guessing

One of the most common attacks

  • Defaults, short passwords, common word searches
  • User info (names, birthday, phone, interests)
  • Exhaustively searching all possible passwords
  • Check by login or against stolen password file
Password Capture

Another angle

  • Watching over shoulder
  • Trojan horse program
  • Monitoring insecure network login (telnet, FTP, web, email)
  • Extracting recorded info after successful login (web history/cache, etc.)
Approaches to Intrusion Detection
Approach 1

Statistical Anomaly Detection

  • Threshold detection — count occurrences of specific event over time; crude & ineffective alone
  • Profile-based — characterize past behavior of users, detect significant deviations; usually multi-parameter
Approach 2

Rule-Based Detection

  • Rule-based anomaly detection — analyze historical audit records to auto-generate rules; no prior knowledge of security flaws required
  • Rule-based penetration ID — uses expert systems technology; rules identify known penetration / weakness patterns
Audit Records & Analysis
Audit Records

Two kinds

  • Native audit records — part of all common multi-user OS, already present, but may not have info in desired form
  • Detection-specific audit records — created to collect wanted info, at cost of additional overhead
Analysis Metrics

What to measure

  • Counter, gauge, interval timer, resource use
  • Tests: mean & standard deviation, multivariate, Markov process, time series, operational
  • Key advantage: no prior knowledge used
Base-Rate Fallacy
Practically, an IDS needs to detect a substantial percentage of intrusions with few false alarms:
  • Too few intrusions detected → false sense of security
  • Too many false alarms → admins ignore / waste time
This is very hard to do. Existing systems do not have a great record.
Distributed Intrusion Detection
Architecture

Agents & central manager

  • Traditional focus is single systems — but typically we have networked systems
  • More effective defence: networked IDS working together
  • Issues: varying audit formats, integrity & confidentiality of data, centralized vs decentralized
  • LAN Monitor & host agents report to Central Manager over WAN
Honeypots

Intrusion Prevention

  • Decoy systems to lure attackers
  • Away from accessing critical systems
  • Collect information on their activities
  • Encourage attacker to stay so admin can respond
  • Filled with fabricated information
  • Instrumented to collect detailed info
  • Single or multiple networked systems
Password Management — Four Strategies
S1

Education

Policies + good user education; guidelines (>6 chars, mixed case + digits + punct, not dictionary words). Often ignored.

S2

Computer-Generated

Let computer create passwords. Random = not memorable → written down (sticky label syndrome). Even pronounceable ones poorly accepted.

S3

Reactive Checking

Periodically run password-guessing tools, disable cracked passwords. Resource-intensive; bad passwords are vulnerable until found.

S4

Proactive Checking

Most promising: allow user to pick, but system verifies acceptability via rules + dictionary of bad passwords + algorithms.

Malicious Software

★ Taxonomy of Malicious Programs
Malicious programs ├─ Needs host program ├─ Independent │ ├─ Trapdoors │ ├─ Worm │ ├─ Logic bombs │ └─ Zombie │ ├─ Trojan horses │ │ └─ Viruses ┐ │ │ ├── Replicate │ └─ (Viruses and Worms also replicate; Worms and Zombies are independent)
The Whole Malware Zoo
Trapdoor

Secret entry point

  • Secret entry point into a program
  • Allows those who know access bypassing usual security
  • Commonly used by developers
  • A threat when left in production programs
  • Very hard to block in OS; requires good s/w development & update
Logic Bomb

Trigger-activated

  • Code embedded in legitimate program
  • Activated when specified conditions met:
    • Presence/absence of some file
    • Particular date/time
    • Particular user
  • Typically damages system — modify/delete files/disks
Trojan Horse

Hidden side-effects

  • Program with hidden side-effects
  • Usually superficially attractive (game, s/w upgrade, etc.)
  • Performs some additional tasks when run
  • Allows attacker to indirectly gain access they do not have directly
  • Often used to propagate a virus/worm or install a backdoor
  • Or simply to destroy data
Zombie

Compromised bot

  • Program which secretly takes over another networked computer
  • Uses it to indirectly launch attacks
  • Often used to launch DDoS attacks
  • Exploits known flaws in network systems
★ Viruses
A piece of self-replicating code attached to some other code (cf. biological virus). Both propagates itself & carries a payload — code to make copies + code to perform some covert task.
Virus Operation — Four Phases
Dormant
⟶
Propagation
⟶
Triggering
⟶
Execution
  • Dormant — waiting on a trigger event
  • Propagation — replicating to programs/disks
  • Triggering — an event triggers the payload
  • Execution — of the payload

Details usually machine/OS-specific — exploiting features/weaknesses.

Virus Structure — Pseudocode
program V := {goto main; 1234567; subroutine infect-executable := {loop: file := get-random-executable-file; if (first-line-of-file = 1234567) then goto loop else prepend V to file; } subroutine do-damage := {whatever damage is to be done} subroutine trigger-pulled := {return true if some condition holds} main: main-program := {infect-executable; if trigger-pulled then do-damage; goto next;} next: }
Types of Viruses
T1

Parasitic Virus

Attaches itself to executable files; traditional form.

T2

Memory-Resident Virus

Lodges in main memory; infects running programs.

T3

Boot Sector Virus

Infects a master boot record; spreads when the system boots from an infected disk.

T4

Stealth Virus

Designed to hide itself from detection by anti-virus software.

T5

Polymorphic Virus

Mutates with every infection — signature-based scanners struggle.

T6

Macro Virus

Macro code attached to a data file (Word/Excel); platform-independent; major source of new infections; blurs distinction between data and program files.

Email Virus
Melissa-style

Spread via email attachments

  • Spread using email with attachment containing a macro virus (cf. Melissa)
  • Triggered when user opens attachment
  • Or worse, even when mail viewed using scripting features
  • Usually targeted at Microsoft Outlook & Word/Excel documents
★ Worms
Definition

Replicating, not infecting

  • Replicating but not infecting program
  • Typically spreads over a network
  • Morris Internet Worm 1988 → led to creation of CERTs
  • Uses user's distributed privileges or exploits system vulnerabilities
  • Widely used by hackers to create zombie PCs for DoS
  • Major issue: lack of security of permanently connected systems (esp PCs)
Worm Operation

Like viruses, plus propagation

  • Dormant
  • Propagation:
    • Search for other systems to infect
    • Establish connection to target remote system
    • Replicate self onto remote system
  • Triggering
  • Execution
Notable Worms
1988

Morris Worm

  • Best-known classic worm — Robert Morris, 1988
  • Targeted Unix systems
  • Propagation techniques:
    • Simple password cracking of local pw file
    • Exploit bug in finger daemon

Firewalls

★ Firewall = a choke point of control and monitoring. Interconnects networks with differing trust, imposes restrictions on network services — only authorized traffic is allowed. Audits and controls access; implements alarms for abnormal behaviour. Must itself be immune to penetration. Provides perimeter defence.
Firewall Limitations
A firewall cannot:
  • Protect from attacks bypassing it (sneaker net, utility modems, trusted orgs, trusted services like SSL/SSH)
  • Protect against internal threats (disgruntled employee)
  • Protect against transfer of all virus-infected programs/files (too many OS & file types)
★ Firewall Types — Four Kinds
Type 1

Packet Filter

  • Simplest of components — foundation of any firewall system
  • Examines each IP packet (no context) and permits or denies according to rules
  • Restricts access to services (ports)
  • Default policies:
    • That not expressly permitted is prohibited (secure default)
    • That not expressly prohibited is permitted (permissive default)
Type 2

Stateful Packet Filter

  • Examines each IP packet in context
  • Keeps track of client-server sessions
  • Checks each packet validly belongs to one
  • Better able to detect bogus packets out of context
Type 3

Application-Level Gateway (Proxy)

  • Use an application-specific gateway / proxy
  • Has full access to protocol:
    • User requests service from proxy
    • Proxy validates request as legal
    • Then actions request and returns result to user
  • Need separate proxies for each service
  • Some services naturally support proxying; others problematic; custom services generally not supported
Type 4

Circuit-Level Gateway

  • Relays two TCP connections
  • Imposes security by limiting which connections are allowed
  • Once created, usually relays traffic without examining contents
  • Typically used when trust internal users by allowing general outbound connections
  • SOCKS commonly used for this

IP Address Spoofing

  • Fake source address to be trusted
  • Add filters on router to block

  • Block source-routed packets

Bastion Host
Definition

Hardened host at perimeter

  • Highly secure host system
  • Potentially exposed to "hostile" elements — hence secured to withstand this
  • May support 2 or more net connections
  • May be trusted to enforce trusted separation between network connections
  • Runs circuit / application level gateways
  • Or provides externally accessible services
Firewall Configurations
Config (a)

Screened Host — single-homed bastion

Packet-filtering router + one bastion host with one NIC. Bastion reachable from the outside, routes inward.

Config (b)

Screened Host — dual-homed bastion

Bastion has two NICs — physical separation of external and internal traffic.

Config (c)

Screened-Subnet Firewall

Outside router + bastion + inside router forming a DMZ. Most secure arrangement.

Case Studies

Case Study 1 — Secure Inter-branch Payment Transactions
General Bank of India (GBI) — ~1200 computerised branches. Electronic Payment System (EPS) with a central server in Mumbai. Branches connect to a local VSAT via dial-up. GBI-Transfer messaging service used for payment instructions. Current security: data is encrypted over a closed network — minimal by modern standards.
Current EPS Architecture — Step by Step
01

Payer branch data entry

Operator enters transaction through EPS interface.

02

Officer validates

Bank Officer checks the validity through EPS.

03

Authorize & store

Authorized transaction stored in local Payment Master (PM) database.

04

Encrypt & queue

Copy encrypted and stored in OUT directory.

05

GBI-Transfer poll

GBI-Transfer polls OUT directory; dials local VSAT; sends files to EPS central office Mumbai.

06

Transit via VSAT

Stored in IN directory at EPS central office.

07

Interface posts to PM

EPS central office interface picks file, sends to PM for processing.

08

Credit request to payee branch

Headers changed; encrypted transaction placed in OUT directory; sent via GBI-Transfer.

09

Payee branch posts credit

PM marks credit, returns acknowledgement.

10

Ack returned to payer

Via EPS central office; transaction complete.

Required Security Enhancements
Enh 1

Non-repudiation

Digital signatures on every transaction.

Enh 2

128-bit Encryption

Upgrade from current 56-bit encryption.

Enh 3

Smart-card Support

Storing sensitive data; on-card digital signing.

Enh 4

Closed-loop PKI

Internal CA & certificate infrastructure.

Proposed solution: Transactions digitally signed and encrypted/decrypted at Payer, Payee, and EPS central office. Signing on the workstation or external smart card. Server side supports automated signing without manual intervention. The flow splits into Payer Leg (Payer → EPS) and Payee Leg (EPS → Payee).
Case Study 2 — Cross-Site Scripting Vulnerability (CSSV)
CSSV (XSS) — a relatively new form of attack exploiting inadequate server-side validation. Attacker's target is not the website, but rather its users (clients/browsers). Happens when malicious tags/scripts attack a Web browser via another site's dynamically generated pages.
Example: www.test.com/address.asp expects user's postal address. User enters: <SCRIPT>Hello World</SCRIPT> Submitted URL: www.test.com/address.asp?address=<SCRIPT>Hello World</SCRIPT> If address.asp echoes the input back into the next Web page without validation, the next page treats the input as script and executes it in the victim's browser. → User would see "Hello World" — or far worse in a real attack.
Case Study 3 — Virtual Elections
Computerized voting must protect individual privacy AND disallow cheating. Simple protocols fail.
Naive Protocol

Broken

  1. Each voter casts vote, encrypts with EA's public key
  2. Voter sends encrypted vote to EA
  3. EA decrypts, tabulates, announces result
Problems: EA doesn't know if votes are from authorized voters or bogus; no mechanism to prevent duplicate voting.
Better Protocol

Still imperfect

  1. Voter casts vote, signs with private key
  2. Voter encrypts signed vote with EA's public key
  3. Voter sends to EA
  4. EA decrypts, verifies signature with voter's public key
  5. EA tabulates and announces result
Remaining issue: EA learns who voted for whom → privacy violation. Needs blind signatures / mix-nets / secret splitting to solve fully.
Secret Splitting (Background)
Original secret: P Random number: R Combined secret: S = P ⊕ R Alice gets only S, Bob gets only R. Either alone is useless. They must come together, combining their shares: P = S ⊕ R → Foundation of secure multi-party protocols, threshold cryptography, and anonymity systems.
